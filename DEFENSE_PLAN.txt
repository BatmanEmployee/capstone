This section defines the minimum architectural components that MUST be present

TIER 1 — CRITICAL (Defense WILL fail without these)
--------------------------------------------------------------------------------

[A] SECURITY ARCHITECTURE
--------------------------

  A-01. bcrypt Password Hashing
        STATUS: MUST VERIFY
        - Every password stored in the DB must use password_hash($pw, PASSWORD_BCRYPT)
        - login_action.php must use password_verify() for new accounts
        - Legacy MD5 demo accounts are acceptable ONLY if documented
        - The panel WILL check your database directly — they will look at the
          password column and ask "what algorithm is this?"
        HOW TO VERIFY:
          SELECT password FROM users LIMIT 5;
          Result should start with "$2y$" (bcrypt) NOT a 32-char hex string (MD5)

  A-02. SQL Injection Prevention (Prepared Statements)
        STATUS: MUST VERIFY ALL FILES
        - Every file in actions/ MUST use $conn->prepare() + bind_param()
        - NO string interpolation like "WHERE id = $id" in SQL
        - pages/*.php files that query the DB must also be checked
        CRITICAL FILES TO AUDIT:
          actions/add_event_action.php
          actions/add_donation.php
          actions/add_distribution.php
          actions/add_announcement_action.php
          actions/add_attendance.php
          actions/add_thread.php
          actions/add_reply.php
          actions/moderate_thread.php
          pages/forum_thread.php (direct query with $thread_id)
          pages/forum.php (direct query with $community_id)

  A-03. CSRF Protection on ALL Forms
        STATUS: MUST VERIFY
        - Every POST form must have <?= csrf_field() ?>
        - Every action file must call csrf_verify() at the top
        - The token must be regenerated after a successful login
        KNOWN MISSING (check these):
          - users.php toggle form
          - Any form added after CSRF was implemented

  A-04. Session-Based Authentication Guard
        STATUS: MUST VERIFY
        - Every protected page MUST start with:
            session_start();
            if (!isset($_SESSION['user_id'])) { redirect to login; }
        - Bypass attempt: panelist will type URL directly in browser while
          logged out — they WILL test this on every page

  A-05. Role-Based Access Control (RBAC)
        STATUS: MUST VERIFY
        - Admin-only pages must check: $_SESSION['role'] === 'admin'
        - Imam/Leader pages must check their respective roles
        - Viewer must be blocked from all write operations
        PAGES THAT MUST ENFORCE ROLE:
          users.php          → admin only
          add_event.php      → admin, imam, leader only
          announcements.php  → admin, imam only (for posting)
          moderate_thread.php → admin, imam only
          add_attendance.php → admin, imam, leader only
        TEST: Login as viewer, manually type URL of admin-only page
              Expected: redirect to dashboard or 403 page

  A-06. Input Validation at Server Side
        STATUS: MUST VERIFY
        - Client-side validation (JS) is nice but NOT security
        - Server-side validation MUST exist in every action file:
            - Name: not empty, max length
            - Email: filter_var($email, FILTER_VALIDATE_EMAIL)
            - Password: minimum 8 characters
            - Role: in_array($role, $allowed_roles)
            - community_id: must be a valid integer > 0
            - event_date: must be a valid date format
        - Registration MUST NOT allow 'admin' role to be selected

[D] ARCHITECTURE PATTERNS
--------------------------

  D-01. Consistent URL/Redirect Structure
        After every POST action, redirect to a relevant page:
          - add_event → redirect to events.php?success=1
          - add_thread → redirect to forum_thread.php?id=X
          - login → redirect to dashboard.php
        NO page should end with a blank white screen after form submit.

  D-02. Flash Message Pattern
        Every success/error message must use the session flash pattern:
          $_SESSION['success'] = "...";  // set in action
          header("Location: ...");
          exit();
          // then in page: show and unset the message
        NO inline echo of errors inside action files.

  D-03. Community Scoping Consistency
        ALL queries that retrieve user-facing data MUST filter by community_id.
        Admin is the ONLY role that sees cross-community data.
        The panel will create a user in Community A, then check if they see
        Community B's data. They WILL test this.

  D-04. Error Handling — No Raw Database Errors Exposed
        NEVER show: "Error: " . $conn->error  to the user
        Acceptable: redirect with generic error message
        Reason: raw DB errors expose schema info — a security violation.

  D-05. Session Timeout
        config/database.php must enforce inactivity timeout (1 hour minimum).
        The panel may ask: "What happens if a user walks away from their desk?"

  D-06. Logout Properly Destroys Session
        logout.php must call session_unset() AND session_destroy()
        Then redirect to login page.
        The panel WILL click back after logout and check if they can re-access
        a protected page.

PART 3: PANELIST QUESTIONS YOU MUST BE READY TO ANSWER
================================================================================

Q1: "Why did you choose PHP over Node.js or Python frameworks?"
    Answer: PHP is the industry standard for XAMPP-based local deployment
    environments. It integrates directly with Apache and MySQL with zero
    configuration, making it accessible for LGU deployment without DevOps
    expertise.

Q2: "How does your system prevent unauthorized access?"
    Answer: Three layers — session-based auth guard on every page,
    role check for sensitive operations, CSRF token on every form.
    Passwords use bcrypt (cost factor 10 default). All DB queries use
    prepared statements to prevent injection.

Q3: "What is your data model for community scoping?"
    Answer: Every user has a community_id. Every community-specific record
    (events, announcements, donations, forum threads, attendance) also stores
    community_id. Queries filter by this ID so users only see their own
    community's data. Admin has a bypass to see all communities.

Q4: "What happens if a user tries to SQL inject your system?"
    Answer: All action files use MySQLi prepared statements with bind_param().
    User input is never concatenated directly into SQL strings.
    Additionally, htmlspecialchars() is used on all output to prevent XSS.

Q5: "How is attendance linked to events?"
    Answer: The attendance table has an event_id foreign key. Each attendance
    record stores: attendee name, present/absent status, who recorded it,
    and which community it belongs to.

Q6: "Can a viewer modify any data?"
    Answer: No. The viewer role has no write access. All action files that
    modify data check the user's role from $_SESSION['role']. Viewers are
    silently redirected if they attempt to access write operations.

Q7: "What is the purpose of CSRF tokens?"
    Answer: Cross-Site Request Forgery tokens prevent attackers from tricking
    an authenticated user into submitting a malicious form. Each session has
    a unique token. Forms include the token as a hidden field. Actions verify
    the posted token matches the session token before processing.

Q8: "Why is bcrypt better than MD5?"
    Answer: MD5 is a hash function designed for speed — it can be brute-forced
    at billions of hashes per second using modern GPU. Bcrypt is a password
    hashing function with an adjustable cost factor that slows down brute
    force attacks to thousands per second. MD5 is also vulnerable to rainbow
    table attacks.

Q9: "What is your testing methodology?"
    Answer: We performed three types of testing:
    1. Functional testing — each module tested per use case (see T3 above)
    2. Security testing — SQL injection, XSS, CSRF bypass, session tests (T4)
    3. Role-based access testing — each role tested for boundary violations (T2)

Q10: "How does the forum ensure community privacy?"
     Answer: Each forum thread stores a community_id. The query on forum.php
     filters by the logged-in user's community_id, so users only see threads
     from their own community. System-pinned threads (community_id = NULL) are
     visible to all as global announcements.
